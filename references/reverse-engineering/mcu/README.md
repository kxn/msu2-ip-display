# MSU2 MCU Firmware Reverse Engineering

目标固件：`MSU2_FirmwareV1.0.hex`  
导出二进制：`MSU2_FirmwareV1.0.bin`（23584 bytes）  
关键反汇编摘录：`key_regions.asm`

分析日期：2026-07-18（第二轮加深）  
范围：MINI 使用场景（通用主控 + 外部 P25D80 资源）

## 2. 运行时状态（gp 相对）

| RAM | gp 偏移 | 角色 |
| --- | ---: | --- |
| `0x200000ca` | `-0x7b6` | **conn**：`0` 离线，`1` 已 MSN 连接 |
| `0x200000c8` | `-0x7b8` | **tick**：Timer ISR 自增；超时掉线 |
| `0x200000bc/bd/be` | `-0x7c4..` | USB/串口 RX 环写/读/长度 |

| `0x200000b4..ba` | `-0x7cc..` | LCD 窗口 x/y/w/h |

| `0x200000de..e1` | `-0x7a2..-0x79f` | USB/显示相关状态位（ISR 路径会清） |
| `0x200000f4` 一带 | `-0x78c` 等 | USB 传输/EP 状态 |
| `0x20000fdc` | `+0x75c` | 大块显示上下文 |
| `0x200002bc` | `-0x5c4` | 启动时 Flash 探测结果等小结构 |

## 3. 主循环 `0x568`

```text
conn = *(u8*)0x200000ca
if conn == 0 → 0x07c4  离线：等 MSNCN / 调 USB 通道维护
if conn == 1 → 在线：RX 组包 → 0x0250 命令分发
超时       → 0x0556  清 conn/tick
```

**重要修正：** 主循环的离线路径 **不负责推进 36 帧动画**。  
离线路径主要是：tick 门限 → 扫 RX 是否为 `MSNCN` → 成功则 `conn=1`；失败则调 `0xe94`（见下）后返回。  
**动图/静态页切换逻辑不在 `0x568` 主循环里**，更可能在：

1. 显示 DMA/定时自动刷屏（启动时配置后自转），或  
2. 尚未钉死的另一条 ISR / 后台任务，或  
3. 与 USB 设备态共存的离线 UI 状态机（散落在 `0x0afc` / `0xc36` / SPI 显示路径）。

这解释了为何很难在主循环里直接看到 `frame++ % 36`。

## 5. SPI Flash 驱动层（加深）

底层字节发送：`0x19d2`（对 `0x40011000` 一带的 SPI/GPIO bitbang 或硬件 SPI 封装）。

### 5.1 已识别的 SPI NOR 命令序列

| 命令字节 | 例程 | 含义 |
| ---: | --- | --- |
| `0x90` | `0x1d2e` | REMS / 读厂商设备 ID；结果 `sh` 到 `gp-0x5c4+4` |
| `0x05` | 状态读 | RDSR |
| `0x06` | WREN | 写使能 |
| `0x03` | `0x1d90` | **标准 Read Data**（addr 24-bit + len） |
| `0x0B` | `0x1ee6` | **Fast Read**（显示/贴图热路径大量调用） |
| `0x20` / `0x52` / `0xD8` | 擦除类 | sector / block erase |
| `0x4B` | `0x1a3e` 一带 | 安全/唯一 ID 类读 |
| `0x48 00 10 00` | `0x1e7c` | **启动时读 256 字节**到 `gp-0x586`，再 `0x1948` 处理 |

`0x48` **不是** JEDEC 标准主命令；对 P25D80 更像厂商扩展读（安全寄存器 / 参数页 / 自定义配置窗）。地址参数为 `00 10 00`（偏移 `0x1000`）。  
注意：镜像里 `0x1000` 正好落在 **第 0 帧彩色图内部**（不是独立 config 页），因此：

- 要么 `0x48` 读的是 **Flash 芯片内部 OTP/SFDP 类空间**（不映射到用户 1MB 线性地址），  
- 要么地址编码与线性 byte address 不同。

`0x1948` 对读回缓冲做校验/混淆式处理（`xori`/`mul` 检查），失败则清零一段表——这是 **启动完整性/屏幕参数校验** 的强信号，但字段尚未符号化。

### 5.2 谁在读用户图像区

| 例程 | 调用方 | 作用 |
| --- | --- | --- |
| `0x1d90` Read | `0x34bc` 等 | 从 Flash 取像素/字形块到 SRAM `0x20000124` |
| `0x1ee6` FastRead | 多处贴图 | 显示热路径 |
| `0x2a6c` | 窗口设置后刷图 | ST7735 窗口 + 送数 |
| `0x25fe` | init | ST7735S 初始化命令流 |

**地址计算**：字形路径里出现 `char - 0xA1`、`* 0x5E`、`+ 0x1270`、`<< 5` 等，是 **GB2312/字库寻址**，不是动图帧号。

### 5.3 尾部目录 `0x0FFE00`：MCU 侧仍未挂上调用链

镜像侧结构（高置信）：

```text
每个 entry 32B：
  +0  u32 LE  flags
  +4  u16 BE  interval_ms   (100 / 1500)
  +6  u16 BE  count         (36 / 1 / 0xFFFF)
  +8  u24 BE  start_page    (0 / 3826 / 3820)
  +20 u16 BE  width
  +22 u16 BE  height
```

| Entry | flags | 语义 |
| ---: | ---: | --- |
| 0 | `0x00000000` | 彩色动图：36 帧 × interval_ms，页 0，160×80 |
| 1 | `0x00020000` (bit17) | 单张彩色：页 3826，160×80 |
| 2 | `0x00000180` (bit7+8) | 单色资源：页 3820，160×68 |
| 3 | `0x00000080` (bit7) | 通配/结束：count=`0xFFFF` |

**flags 很像资源类型位图**（彩色/单色/多帧/单帧），这正是“不同屏厂资源包可共用主控”的接口。

**缺口：** 在 V1.0 hex 里 **尚未找到** 构造地址 `0x0FFE00` 的 `lui/addi` 或对这 128 字节的显式解析循环。可能原因：

1. 地址由 **运行时从表计算**（表在 `0x48` 读出的参数里）；  
2. 目录由 **显示硬件/DMA 描述符** 间接引用；  
3. 实机固件相对 V1.0 有小改（未读回则无法证伪）。

## 6. 启动序列里和 Flash 相关的步骤

```text
0x386a  0x1a88   GPIO/时钟类 init
0x386e  0x1d2e   SPI 0x90 读 Flash ID → 存 gp-0x5c4+4
0x3876  0x1928   拷贝/准备默认表
0x387a  0x1a28   更多外设
0x387e  a0 = gp-0x586
0x3882  0x1e7c   SPI 0x48 读 256B → 缓冲
0x3886  0x1948   校验/解析缓冲（失败清表）
0x3890  置启动完成标志 0x200000cb = 1
0x389a  0x25fe   ST7735S init
0x38ae  0x2b54   清屏/窗口
0x38cc  0x2fc4   再刷一块
0x38d0  0x27b8   另一路外设（USB/定时）
→ 进入 0x568 超级循环
```

结论：启动时 **确实从 Flash（或芯片扩展空间）拉配置**，但“屏参”具体字段还没符号化；用户可见的 1MB 布局差异主要仍在 **资源 + 尾部目录**。

## 7. 离线 Mode / 按钮 / Logo（实机对齐）

### 7.1 目录 entry = 离线 mode 槽

| 索引 | 页 | 出厂内容 | 实机 mode |
| ---: | ---: | --- | --- |
| E0 | 0 起 36 帧 | 待机动画 | **动图 mode** |
| E1 | **3826** | CLK_BG | **静图 mode**（用户确认：刷机后为「获取 IP 中」） |
| E2 | **3820** | MLOGO 160×68 mono | **Logo 资源**；是否进按钮轮换待确认 |
| E3 | — | 通配 | 非独立画面 |

### 7.2 按钮逻辑（行为级，非单指令）

```text
conn==0:
  触摸 → 切换当前 entry（至少 E0 ↔ E1）
  显示函数按 entry.page / count / interval / flags 工作
conn==1:
  触摸离线 UI 不生效；host 命令占用 3826/3926 等
```

### 7.3 开机 logo（规格）

| 项 | 值 |
| --- | --- |
| 页 | **3820..3825**（E2） |
| 尺寸 | **160×68** |
| 格式 | 1bpp，行 20B，MSB 左，1360B 有效 + 176B 零填充 = 1536B |
| 颜色 | 1 位资源；显示时上色。V1.0 启动路径前景≈`0x001F` 蓝、背景白 |
| 出厂点阵 | `mlogo_160x68_factory.mono`（从京东方镜像截取） |
| 新设计参考 | `mlogo_160x68_design.mono` + `docs/mockups/msu2-boot-logo-preview.png`（**未改 flasher**） |

启动 `0x2fc4` 另有硬编码页 3779（MINI 出厂近空）。有内容的 logo 以 **3820** 为准。  
`dhcp_failed` 写 `3726..3825` 会抹掉 logo。

### 7.4 flasher 与离线 mode 冲突（根因）

| flasher 常量 | 页 | 撞 |
| --- | ---: | --- |
| `ACQUIRING_PAGE=3826` | 3826 | **E1 离线静图** |
| `DHCP_FAILED_PAGE=3726` | 3726..3825 | **E2 logo 3820..3825** + ASC64 尾 |

用户现象「静图 = 获取 IP 中」是 **页复用**，不是 mode 机坏了。

| 基址 | 角色 |
| --- | --- |
| `0x40011000` | SPI 字节级收发（Flash + 可能共线） |

## 9. 外部 Flash 里“还可能有价值”的参数清单

### 已实锤（镜像级）

1. **动图 interval_ms**（尾部 E0 `+4`）  
2. **动图帧数**（E0 `+6`，通常 36）  
3. **各资源起始页 / 宽高**（E1/E2…）  
4. **flags 类型位**（彩色/单色/多帧等）——资源类型接口  
5. **屏厂面板描述块 @ `0x0FFF00`**（见 §9.1，翰彩完整 1MB 镜像）  

### 高可疑、待符号化（MCU 启动读）

6. **`0x48` 读回 256B**（`0x1e7c`→`0x1948`）：仍可能叠一层校验/备用参数  
7. **Flash ID**（`0x90`）是否分支不同驱动  
8. **字库基址/字符映射表**  

### 低价值/已排除

- 资源区（0..约 4044 页）在京东方/翰彩 **字节级完全相同**  
- `Flash_Info` 字符串 = MSN SFR 名，不是布局  

### 9.1 京东方 vs 翰彩镜像 diff（2026-07-18）

| 文件 | 大小 | 说明 |
| --- | ---: | --- |
| `Flash_MINI_V1.1(京东方屏幕).bin` | `0xFFE80` (1048192) | 资源 + 128B 目录后截断 |
| `Flash_MINI_V1.1(京东方屏幕)低速切换.bin` | 同上 | 仅目录 interval 100→1500 |
| `Flash_MINI_V1.1(翰彩屏幕).bin` | `0x100000` (1048576) | **完整 1MiB** |

**重叠区 `0x000000..0x0FFE7F`：与京东方 V1.1 完全一致（0 字节差异）。**  
差异 **只在** 翰彩多出来的尾部：

```text
0x0FFE00..0x0FFE7F  资源目录（两家相同）
0x0FFE80..0x0FFEFF  0xFF 填充（128B）
0x0FFF00..0x0FFF77  ★ 面板配置块（约 120B 有效 + 后随 FF）
0x0FFF78..0x0FFFFF  0xFF 填充
```

#### 面板配置块 @ `0x0FFF00`（翰彩）

| 偏移 | 长度 | 值 / 示例 | 含义（高置信） |
| ---: | ---: | --- | --- |
| +0x00 | 2 | `00 01` | 版本 = 1 |
| +0x02 | 1 | `2A` | CASET 命令码 |
| +0x03 | 1 | `2B` | RASET 命令码 |
| +0x04 | 1 | `2C` | RAMWR 命令码 |
| +0x05 | 1 | `36` | MADCTL 命令码 |
| +0x06 | 2 | `FF FF` | 分隔 |
| +0x08 | 8 | `"ST7735\0\0"` | 面板/驱动名 |
| +0x10 | 2 | `00 A0` | 宽 160（BE） |
| +0x12 | 2 | `00 50` | 高 80（BE） |
| +0x14 | 1 | `01` | **列偏移 colstart = 1** |
| +0x15 | 1 | `1A` | **行偏移 rowstart = 26**（0.96" 160×80 常见） |
| +0x16 | 2 | `FF FF` | 分隔 |
| +0x18 | 4 | `78 A8 00 00` | 待定（延迟/附加标志） |
| +0x1C | 4 | `00 00 00 00` | 保留 |
| +0x20 | ~88 | `62 11 B1 05 3C…` | **LCD init 命令流**（厂商时序） |

这直接证实：

- **屏厂差异参数放在 Flash 末尾固定槽**，不改资源、不改主控 hex 即可换屏驱动表。  
- 京东方发布包 **截断掉了** 这块（或该屏走 MCU 内硬编码 init `0x25fe` 那条 ST7735 序列）。  
- 资源目录 + 动画/字库与屏厂无关；**换屏只换 `0xFFF00` 面板块（外加可能的偏移）**。

Init 流与 MCU 内硬编码 init（`0x25fe` 的 `E0/E1` gamma 等）**不是同一套**，进一步说明 Flash 表可覆盖/替换默认初始化。

### 对“不同厂家屏幕”的含义（已用两份 bin 验证）

- **同一主控 + 同一资源**，差异集中在 **`0x0FFF00` 面板描述**。  
- 发布包长度不同：京东方 `0xFFE80`、翰彩满 `0x100000`。  
- 做完整恢复工具时：翰彩必须写满 1MB（含面板块）；京东方文件写到 `0xFFE80` 即可，但若实机是翰彩模组，缺面板块可能导致显示偏移/时序不对。
