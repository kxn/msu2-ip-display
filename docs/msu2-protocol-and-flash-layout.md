## 硬件与两套“固件”

MSU2 系列有两块独立存储，不要混谈：

| 部件 | 型号 / 规格 | 作用 | 更新方式 |
| --- | --- | --- | --- |
| 主控 MCU | 沁恒 **CH32X033F8P6**（QingKe RISC-V，同系列 CH32X035 文档常用 62KB Flash / 20KB SRAM） | 通用协议/状态机、LCD 与 SPI Flash 驱动 | WCHISPTool / WCHISPStudio，USB 下载 |
| 外部 SPI Flash | **P25D80**，**1024KB**，页大小 **256B**，共 **4096** 页（页号 `0..4095`） | **产品差异化数据**：图片、字库、数码管、资源目录/播放参数 | 上位机串口协议“闪存固件更新 / 局部写页” |
| LCD | MINI：**ST7735S**，**160×80** | 显示 | 主控驱动（具体时序/分辨率更可能由固件 + Flash 侧配置共同决定） |

进入主控 USB 烧录模式的官方方法：短接 **PC17/USB_D+** 与 **3.3V** 后上电，插入 USB 1~3 秒后松开；WCH 工具里选 `CH32X03X` / `CH32X033F8P6`，关闭 RST 复位功能。

### 工作模型（当前最佳假设）

官方材料更符合 **「主控通用固件 + 外部 Flash 放屏参/资源」**：

- 指南写明 MSU2 与 MSU2_MINI **同一主控**、同一套主控烧录流程，没有“按屏幕尺寸分发不同 hex”的说明。
- 产品差异（0.96/1.54、动画、字库、背景、帧间隔）集中在外部 Flash 镜像；MINI 侧明确提供 `Flash_MINI_V1.1(...).bin`。
- 外部 Flash 尾部 128B 已能解析出资源页指针与动图间隔（正常 100ms / 低速 1500ms），这是典型的 **参数外置**。
- MCU hex 内有 `Flash_Info` / `MSN_UID` / `MSN_Status` 等符号串，并含 `MSNCN` / `MSN01` 协议串；布局页号如 `3826/3926/3651` **未见大量硬编码常量**（至少不是简单立即数铺开），与“运行时读 Flash 目录/资源”一致。

因此仓库中的 `MSU2_FirmwareV1.0.hex` 应视为 **系列通用主控固件的官方发布物**，逆向它是合理主路径。

仍需保留的版本差：

| 对象 | 仓库/命名 | 含义 |
| --- | --- | --- |
| 主控 hex | `MSU2_FirmwareV1.0.hex` | 资料包标注 **V1.0** |
| 外部 Flash | `Flash_MINI_V1.1(京东方屏幕).bin` | 资源包标注 **V1.1**（板级/屏厂/资源迭代） |
| 实机 | 当前 MINI 硬件表现像 1.1 资源世代 | **主控是否仍为 V1.0 字节级一致：未读回验证** |

`Flash_MINI_V1.1(...).bin` **不是** MCU 程序。改动画/背景/字库/帧间隔优先改外部 Flash；改命令集、握手状态机、驱动策略等仍要动 MCU 逻辑（以 V1.0 hex 为基线补丁，或确认实机版本后再改）。

## 已实机确认的 MINI flasher 参数

- USB VID/PID: `1A86:FE0C`
- Windows 端口样例：`COM4`
- 串口：`921600 8N1`，RTS/CTS 硬件流控
- 握手：`00 4D 53 4E 43 4E`，即 `\0MSNCN`
- LCD: `160x80`
- 彩色图格式：RGB565 big-endian
- 单张 `160x80` 彩色图大小：`160 * 80 * 2 = 25600` bytes
- Flash 页大小：`256` bytes
- 单张 `160x80` 彩色图占 `100` 页

之前的硬件验证在 `COM4` 上完成了 `3800/3800` 页写入，并在最后发送 page 0 预览命令，回包匹配。

## 官方资料修正点

早期逆向验证曾使用 `19200 8N1` 无流控，设备也能握手；但对照官方 1.47 寸和 2.8 寸 Python 参考代码后，已在当前 MINI 上重新确认 `921600 8N1 + RTS/CTS` 可以握手并工作。当前 flasher 因此使用高速路径。

官方参考代码实际打开串口时使用：

```text
921600 baud, xonxoff=False, rtscts=True
```

注意：部分 Demo 结构体默认字段仍写着 `baud_rate=19200`，那是历史残留；真实 `serial.Serial(...)` 调用以 `921600 + rtscts=True` 为准。未来 `host-usb/` 不要把 `19200` 写死成全系列唯一参数。

## 握手流程

官方 Demo 会等待设备周期性输出类似 `00 4D 53 4E 30 31` 的 `\0MSN01` 版本广播，再由上位机发送：

```text
00 4D 53 4E 43 4E
```

设备确认连接时会回：

```text
00 4D 53 4E 43 4E
```

当前 flasher 的握手逻辑允许回包缓冲里夹带设备广播，只要包含 `\0MSNCN` 即认为成功。

## Flash 命令

Flash 总容量按官方文档为 `1024KB`，即 `4096` 页，每页 `256` bytes。

擦除是把 bit 置 1；写入只能把 1 改成 0。写任何页前必须先擦除目标区域。

### 擦除页

```text
03 02 start_hi start_lo count_hi count_lo
```

含义：

- `03`: Flash 操作
- `02`: 按页擦除
- `start`: 起始页，16-bit big-endian
- `count`: 页数，16-bit big-endian

示例：擦除 `3826` 开始的 `100` 页：

```text
03 02 0E F2 00 64
```

### 写入页缓冲

一页 256 bytes 先拆成 64 组，每组写 4 bytes：

```text
04 index data0 data1 data2 data3
```

`index` 范围是 `00..3F`。

### 提交一页写入

当前 flasher 使用快速写入形式，64 组 `04` 包后接：

```text
03 03 page_hi page_mid page_lo 01
```

含义：

- `03`: Flash 操作
- `03`: 快速写 Flash
- `page`: 24-bit big-endian 页号
- `01`: 写 1 页

官方旧 Demo 里还有普通写入形式：

```text
03 01 page_hi page_mid page_lo page_count
```

当前 flasher 没有使用普通写入。

### 读 Flash 字节

官方 Demo 的读字节命令：

```text
03 00 addr_hi addr_mid addr_low 00
```

当前 flasher 不需要读 Flash；完整恢复工具或逆向布局时会用到。

## LCD 显示命令

设置显示起点：

```text
02 00 x_hi x_lo y_hi y_lo
```

设置显示尺寸：

```text
02 01 width_hi width_lo height_hi height_lo
```

显示彩色图片页：

```text
02 03 00 page_hi page_lo 00
```

实测注意：显示普通彩色 flash 页前需要先设置显示窗口，例如全屏页应先发送
`02 00 00 00 00 00` 和 `02 01 00 A0 00 50`，再发送上面的页显示命令。
flasher 的 `preview_pages` 也是这个顺序；host-usb 的 pending/DHCP 状态页也必须使用同样顺序。

早期 flasher 验证流程曾依次显示 page `0`、`3826`、`3926`，最后回到 page `0`；当前正常写入流程不再自动预览 page `0`，`preview_pages` 只保留为调试/实验辅助。

host-usb 侧已用到的扩展显示命令（来自官方 Demo / 实机探测）：

| 命令 | 含义 |
| --- | --- |
| `02 02 fg_hi fg_lo bg_hi bg_lo` | 设置前景/背景色（RGB565） |
| `02 03 07 00 00 00` | 装载 LCD 写地址（随后用 `04` 缓冲 + `02 03 08 ...` 直写像素） |
| `02 03 0d fill 00 00` | RAM 初始化 / 填充 |
| `02 03 0f addr_hi addr_mid addr_lo` | 带 mask 把 Flash 地址内容叠加到 RAM（数码管字形常用） |
| `02 03 11 addr_hi addr_mid addr_lo` | 以指定彩色背景页混显并上屏 |

IP 数码显示典型路径：背景页 `3926` + 数字点阵页 `4026 + digit`，通过 RAM mix 叠加，而不是重写整张彩色图。

官方 Demo 还包含单色图片显示、彩色背景叠加、直接写 LCD 数据、SFR/ADC/RGB LED 等接口。完整上位机能力应从官方 Python API 继续整理。

### 镜像尾部资源目录（关键发现）

偏移 `0x0FFE00` 起 128 字节，可按 **4 条 × 32 字节** 解释。字段尚属启发式还原，但与资源页、分辨率高度吻合：

```text
+0x00  u32 LE  flags / 类型
+0x04  u16 BE  帧间隔相关（见下）或页跨度类参数
+0x06  u16 BE  帧数 / 计数
+0x08  u24 BE  起始页（高字节常为 0x00，后两字节为页号）
+0x0B..          填充 0xFF
+0x14  u16 BE  宽度
+0x16  u16 BE  高度
+0x18..          填充 0xFF
```

| Entry | flags | 关键语义 | 页 | 尺寸 | 备注 |
| ---: | ---: | --- | ---: | --- | --- |
| 0 | `0x00000000` | 离线动图表头 | 0 | 160×80 | 正常版 `u16BE@+4 = 0x0064`（100）；低速版 `0x05DC`（1500） |
| 1 | `0x00000200` | 单张彩色背景 `CLK_BG` | **3826** | 160×80 | `count=1`，与 100 页图一致 |
| 2 | `0x00000180` | 单色 Logo `MLOGO` | **3820** | **160×68** | 与 6 页单色 Logo 吻合 |
| 3 | `0x00000080` | 占位/通配 | 0 | 160×80 | `count=0xFFFF`，含义未完全确认 |

**两版固件唯一差异：**

| 偏移 | 正常版 | 低速切换版 | 解释 |
| --- | ---: | ---: | --- |
| `0x0FFE04..05` | `00 64` | `05 DC` | big-endian `100` vs `1500` |

结合官方文档“离线每 **100ms** 切一帧”，`0x0064=100` 几乎可确定是 **动图帧间隔（毫秒）**；低速版把间隔改成 **1500ms**，资源像素本身不变。  
**改播放速度不必重刷 900KB 动画，只改尾部目录 2 字节即可**（仍建议整镜像校验后写入）。

MCU 离线状态机是否完全只读这份目录、还是固件内也有硬编码 100ms，尚未对 MCU 反汇编确认；但“低速切换”镜像只改此处，强烈暗示目录项会被使用。

## 离线 mode、开机 logo 与 flasher 覆盖（实机结论）

### 尾部目录 = 离线可播放资源表（高置信）

`0x0FFE00` 四条 entry 就是固件离线 UI 能“点名播放”的资源槽：

| Entry | flags | interval | count | 起始页 | 尺寸 | 官方意图 | 被本仓库 flasher 写成 |
| ---: | ---: | ---: | ---: | ---: | ---: | --- | --- |
| **E0** | `0x00000000` | 100ms（低速包 1500） | **36** | **0** | 160×80 彩色 | **离线动图** | `offline_animation`（未连接闪烁） |
| **E1** | `0x00020000` | 100 | **1** | **3826** | 160×80 彩色 | **离线静图 / 时钟背景 CLK_BG** | **`acquiring`「获取 IP 中」** |
| **E2** | `0x00000180` | 100 | 1 | **3820** | 160×68 **单色** | **MLOGO 开机/品牌 logo** | 落在 DHCP 失败写入带 `3726..3825` 内 → **整段被盖掉** |
| **E3** | `0x00000080` | 100 | `0xFFFF` | 0 | 160×80 | 结束/通配 | 不占用页 |

### 按钮切换 mode（无 host 连接时）

实机 + 目录交叉结论：

1. **仅 `conn==0`（未 MSN 握手）时按钮有效**；host 连上后主循环只跑命令分发。
2. 至少 **2 个离线视图**（你已实机看到）：
   - **Mode A — 动图**：播 E0，页 `0,100,…,3500`，默认 100ms/帧。  
   - **Mode B — 静图**：播 **E1 单帧页 3826**。  
     你确认静图就是 **「获取 IP 中」** → 与 flasher 把 `ACQUIRING_PAGE=3826` 写成 acquiring 图 **完全一致**。  
     **不是固件 bug**：离线静图槽和 host「获取中」背景 **共用 3826**；出厂是 CLK_BG，被 flasher 换成了获取中。
3. **Mode C（可能）— 单色 logo**：E2 页 3820，160×68 mono，点阵内容可辨 “MSU2” 类 logo。  
   若第三下按钮有“小 logo / 偏上显示”，就是它；若只有两态，则 E2 可能只用于开机绘制。
4. MCU 主循环里 **没有** `frame++%36`；动图推进多半是 **读目录 interval + 定时/DMA 刷 E0**。按钮逻辑应是 **切换当前 entry 索引（0↔1↔?）**，再按 entry 的 page/count/flags 显示。入口尚未在反汇编里钉到单条指令，但 **行为与目录表一一对应**，利用方式已足够。

```text
未连接:
  [触摸] → 切换 offline_view_index
           0: 播 E0 动图 (page 0..)
           1: 显示 E1 静图 (page 3826)  ← 你现在看到的「获取 IP 中」
           2?: 显示 E2 mono logo (page 3820) 若存在第三态

已连接 (MSNCN):
  按钮离线 UI 被挡住；3826/3926/3726 由 host 命令解释
```

### 为何静图变成了「获取 IP 中」（你的发现）

| 写入 | 页范围 | 撞上什么 |
| --- | ---: | --- |
| flasher `acquiring` | **3826..3925** | **离线静图 Mode B（E1）整槽** |
| flasher `dhcp_failed` | **3726..3825** | ASC64 尾 + **MLOGO 3820..3825** |
| flasher `ip_bg` | 3926..4025 | 在线 IP 背景（E 表未直接当离线静图） |
| flasher 动图 | 0..3599 | E0 动图（有意替换） |

所以：

- 未连接 → 按按钮 → 静图 mode → 读 **3826** → 显示 flasher 写的 **获取中**。  
- 出厂未刷时，同一 mode 应是官方 **CLK_BG**。  
- **若希望离线静图 ≠ 获取中**：静图专用页不要用 3826，或 3826 只留给离线、host 获取中改到空洞页（如 3600 一带），并接受改 host 协议常量。

### 你能怎么利用（不改 MCU 的前提下）

| 目标 | 做法 |
| --- | --- |
| 自定义离线动图 | 写 E0：`0..3599`（已在做） |
| 自定义离线静图（按钮第二态） | 写 **3826** 一张 160×80；**不要**再拿它当「仅 host 获取中」除非你接受共用 |
| 自定义 logo | 按上表规格写 **3820** 起 6 页；**不要**让 `3726..3825` 整段擦写盖住 |
| 调离线动画快慢 | 改目录 E0 `interval`（`0x0FFE04`：100 / 1500） |
| 关掉某一离线态 | 改目录 count/page（例如 E1 count=0 或 page 指到与 E0 相同）——需实机试 |
| Host 状态图不抢离线槽 | `acquiring`/`dhcp_failed` 迁出 3826/3820 占用区 |
| 按钮当“页切换器” | 把 E0/E1/E2 做成三套画面，触摸在离线时轮换（E2 是否进轮换以实机为准） |

### Flash 末尾「逻辑向」可配项（只关心固件行为）

```text
0x0FFE00  资源目录 4×32B
          ├─ 播什么（page）
          ├─ 播几帧（count）
          ├─ 多快（interval_ms）
          ├─ 彩色/单色等（flags）
          └─ 宽高（与裁剪/居中有关）
0x0FFE80  填充
0x0FFF00  面板块（翰彩满镜像才有）：ST7735 名、160×80、col/row 偏移、init 流
          └─ 影响「怎么点亮屏」，不是 mode 文案
```

**不能**靠 Flash 配置改的：串口协议、握手、host 命令表、按钮是否在在线时可用（在线被 `conn` 挡住）。

### 建议的布局修正（若你要同时保留离线静图 + host 状态）

示例（需改 `assets.rs` / host 常量并实机确认 E1 是否必须 3826）：

| 用途 | 建议页 | 理由 |
| --- | ---: | --- |
| 离线动图 | 0..3599 | E0 |
| 离线静图（按钮） | **3826** 专用美图 / 或改目录指到 3600 | 现 E1 |
| Logo | **3820** 保留 6 页 | E2 |
| Host 获取中 | **3600** 或其它空洞 | 避开 E1 |
| Host DHCP 失败 | **3700** 附近空洞，**不要盖到 3820** | 现 3726 太靠 logo |
| Host IP 背景 | 3926 | 可保留 |

更稳的长期做法：刷机时 **重写 `0x0FFE00` 目录**，让 E1 指向你的静图页，E2 保持 logo，host 状态页全部放在目录未引用的空洞里。

### 当前项目紧凑布局（已采用）

当前 flasher 已采用“重写资源目录 + 两帧离线动图”的紧凑布局。资源目录 page `4094` 由 PC 端基于官方 MINI 目录模板生成完整 256-byte 页，再按普通 Flash 页流程擦除并整页写回；不是在设备端局部 patch 几个字节。

| 用途 | 页 | 备注 |
| --- | ---: | --- |
| 离线动图可见帧 | `0..99` | E0 frame 0 |
| 离线动图空白帧 | `100..199` | E0 frame 1 |
| 离线静图 | `200..299` | E1 指向这里，按钮静图显示不闪烁的“未连接” |
| Host 获取 IP 中 | `300..399` | host-usb `PENDING_PAGE` |
| Host DHCP 失败 | `400..499` | host-usb `DHCP_FAILED_PAGE` |
| Host IP 背景 | `500..599` | host-usb `IP_BACKGROUND_PAGE` |
| 启动 logo | `3820..3825` | E2，160x68 mono |
| 数码管字形 | `4026..4037` | 保留官方 N24X33P，host 使用 `4026 + digit` |
| 资源目录 | `4094` | E0/E1/E2/E3 table |
| Panel config | `4095` | 保留，不写入 |

当前资源目录字段：

| Entry | interval | count | start page | 说明 |
| ---: | ---: | ---: | ---: | --- |
| E0 | `900ms` | `2` | `0` | 两帧闪烁离线动图 |
| E1 | `100ms` | `1` | `200` | 离线静图 |
| E2 | `100ms` | `1` | `3820` | 启动 logo |
| E3 | 保持官方值 | 保持官方值 | 保持官方值 | 占位/结束 entry |

`3726`、`3826`、`3926` 是旧布局值。新代码不应再使用这些页号；它们只应出现在历史说明、厂商/出厂布局说明或调试对比里。


## 完整 1MB 恢复 vs 局部刷写

| 场景 | 建议 |
| --- | --- |
| 砖了 / 想回出厂动画与字库 | 用官方 Demo“闪存固件更新”刷整份 `Flash_MINI_V1.1(...).bin`（约 10 秒级） |
| 只换待机动画 | 擦写 `0..3599`（36×100 页）；可选同步改尾部间隔 |
| 只换时钟/IP 背景 | 擦写 `3826..3925` 和/或 `3926..4025` |
| 只改离线切帧速度 | 改镜像尾部 `0x0FFE04` 的 100/1500，或后续确认 MCU 参数后再改 |
| 本仓库 flasher 现状 | 局部写 8 个资源（两帧离线动图、离线静图、DHCP、获取中、IP 背景、logo、资源目录），保留 `4026+` 和 page `4095` |

整片恢复时注意：官方 bin 长度为 `0xFFE80`，不是严格 `0x100000`；写入工具应按文件实际长度或按页 `0..4093` + 尾 128B 处理，不要假设尾部未给出区域有有效数据。

## 资源格式速查

| 类型 | 计算 | 页数 |
| --- | --- | ---: |
| 160×80 彩色 RGB565 | `160*80*2=25600` | 100 |
| 160×80 单色 | `ceil(160/8)*80=1600` | 7 |
| 160×68 单色 | `20*68=1360` | 6 |
| 32×64 ASCII 单字 | `32*64/8=256` | 1 字/页 |
| 128 字符 ASC64 字库 | `256*128=32768` | 128 |
| 24×33 数码管单字 | 约 `ceil(24/8)*33=99`，按页对齐存放 | 官方按 12 页整包 |

彩色取模：Image2Lcd，16 位色，**高字节在前（big-endian）**。小端写入会花屏/暗红横线（已验证）。

单色取模：1bpp，行宽按整字节对齐，宽度宜为 8 的倍数。

## 芯片 / MCU 逆向相关

### 主控固件（以 V1.0 hex 为逆向基线）

- 芯片：`CH32X033F8P6`；运行时 USB 串口枚举 `1A86:FE0C`。
- 片内 Flash 量级约 **62KB**，SRAM 约 **20KB**（CH32X035/033 系列公开资料；以数据手册为准）。
- 仓库 `MSU2_FirmwareV1.0.hex`：约 **23.5KB** 有效映像（`0x0000..0x5C20`）。
- 在「通用主控 + Flash 资源/参数」模型下，这份 **V1.0 hex 就是当前最应逆向的正式主控固件**。
- 与板载 **Flash_MINI V1.1** 的关系：版本号差在 **外部资源包**，不自动等于主控也升到了 1.1；实机主控可能仍是 V1.0 逻辑，仅资源为 V1.1。
- **读回限制**：官方 WCHISPTool / WCHISPStudio 常规流程是下载/校验/擦除，**通常没有面向用户的 Code Flash 整片读出**；开源 [wchisp](https://github.com/ch32-rs/wchisp) 也明确以 flash/verify/config 为主，**不提供主程序 Flash dump**。代码保护打开时更无法读出。因此实践上 **只能以仓库这份 hex 为基线做静态逆向 + 串口行为对照**，而不是先 dump 再比。
- 更新主控仍靠 ISP 短接 + WCH 工具；改错需能重新下载这份（或你改过的）hex 救砖。

### 外部 Flash 镜像（本仓库已具备）

- 已完整可分析：动画帧、字库、背景、数码管、尾部目录。
- 可直接改资源、换图、调帧间隔（目录字段）——这是产品差异的主战场。
- **不能**单独靠它改协议状态机、命令解析等 MCU 逻辑；但很多“功能不够”其实用 host 协议 + 改资源就能补。

### 若要改“逻辑”，现实路径

1. **优先不改 MCU**：host-usb / flasher 用现有串口 API 做 IP、状态图、直写 LCD、局部刷图。
2. **只改资源/参数**：重打包或局部写 `Flash_MINI_*.bin`；帧间隔先改尾部 `100/1500` 并实机验证。
3. **改 MCU（基线 = `MSU2_FirmwareV1.0.hex`）**：
   - 静态反汇编 QingKe RISC-V；对照 CH32X035/033 头文件与串口抓包。
   - 重点找：命令分发（`02`/`03`/`04`/`00 MSN…`）、读 `Flash_Info`/尾部目录、离线 36 帧循环、LCD 初始化。
   - 产出补丁 hex 后用 ISP 下载；保留原版 V1.0 与外部 Flash 镜像以便回滚。
   - 风险：USB/串口/屏驱/Flash 驱全在 MCU；体量小（~24KB 有效）使静态逆向可行。

### 当前仓库逆向材料边界

| 材料 | 能回答什么 | 不能回答什么 |
| --- | --- | --- |
| `Flash_MINI_V1.1*.bin` | MINI 资源布局、出厂图、帧间隔目录 | MCU 代码路径 |

## 当前不确定项

- `921600 + RTS/CTS` 已在当前 MINI 上确认可握手；其他型号/旧资源包是否都适用，仍需按设备确认。
- 尾部目录 32 字节记录的完整字段语义（尤其 entry3 的 `0xFFFF`、flags 位域）仍是启发式，应用前最好用改字节 + 实机离线播放确认。
- MCU 是否 **只** 从尾部目录读帧间隔，还是内部仍有默认 100ms，需在 V1.0 hex 反汇编中确认。
- 实机主控与 `MSU2_FirmwareV1.0.hex` 是否字节级一致：在无法 dump 的前提下，只能用行为对照（协议、离线策略、异常路径）间接验证。
- 完整上位机 API 还包括 SFR 读写、ADC、RGB LED、截图投屏等；本文只覆盖 flasher / host 显示 / Flash 布局最需要的部分。
- 当前内置 RGB565BE 资产不是官方出厂图；完整“出厂级”资源恢复请用 `Flash_MINI_V1.1(...).bin`。

## MCU 逆向进展（V1.0 hex）

详细笔记：`references/reverse-engineering/mcu/README.md`  
关键反汇编：`references/reverse-engineering/mcu/key_regions.asm`  
扁平二进制：`references/reverse-engineering/mcu/MSU2_FirmwareV1.0.bin`

### 已确认

| 点 | 结论 |
| --- | --- |
| 超级循环 | `0x37b8` 初始化后，`0x38e0` 反复调用 `0x568` |
| 连接标志 | SRAM `0x200000ca`：`0` 离线，`1` 已 MSN 握手 |
| 在线行为 | `conn==1` 时走 RX 组包 + `0x0250` 命令分发；**host 命令接管** |
| 握手 | 离线路径比对 `MSNCN`，成功则 `conn=1` |
| 掉线 | Timer ISR 递增 `0x200000c8`；超时调用 `0x556` 清 `conn` |
| 命令表 | `00`/`02`/`03`/`04`/`08` 与官方 Python 一致 |
| SPI Flash | 识别 `03` Read、`0B` FastRead、`90` ID、擦除类；显示贴图走 FastRead |
| 启动配置读 | `0x1e7c` 发 `0x48 00 10 00` 读 256B → `0x1948` 校验/解析（屏参/完整性的高可疑点） |
| 布局立即数 | MCU 内几乎无 `3826/3926` 硬编码；尾部目录在镜像侧已解析，**MCU 加载点尚未挂上** |
| `Flash_Info` 字符串 | MSN SFR 属性名，不是布局表 |
| USB | `0x40023000` 为 USBFS；描述符产品名 `CH32x035` |

### 触摸 / Mode（第二轮更谨慎）

- 官方文档 **未描述** 触摸切换；仅有“离线 36 帧动图 / 在线 host 改背景”。
- 实机“动图 ↔ 静态”更像在 **资源 entry 之间切换**（E0 动图 vs E1/E2 静态页），不是文档里的正式 mode API。
- **`0xe94` 已纠正**：是 USB 通道维护，不是显示 mode 枚举。
- **主循环不推进帧动画**；动图引擎应在 DMA/定时/隐藏任务里，仍在追。
- 禁用触摸：在定位到“切换当前 entry/基址”的指令前，最稳仍是 **host 保持连接**。

### 外部 Flash 仍可能有价值的参数

| 优先级 | 内容 | 状态 |
| --- | --- | --- |
| 高 | 尾部目录 interval/count/page/flags | 镜像已解析 |
| 高 | **`0x0FFF00` 面板配置**（名/分辨率/偏移/init） | 翰彩 bin 已解析头部；init 流待完全符号化 |
| 中 | 启动 `0x48` 读回 256B | 有代码路径，字段未符号化 |
| 低 | 资源区屏厂差异 | **无**（京东方=翰彩） |

### 建议的后续工作

1. 对照 P25D80 手册把 `0x48`/`0x1948` 字段符号化。  
2. 找 tail 目录加载与 interval 比较点（动图时钟）。  
3. 找触摸边沿 → 切换 entry 的代码。  
4. 若有另一厂家 `Flash_*.bin`，优先 diff 尾部 128B。  
5. 功能优先 host + 外部资源；MCU 补丁留到入口钉死后。
